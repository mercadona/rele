# rele — Claude Code ground rules

Relé is a Python library that makes Google Cloud Pub/Sub easy: declarative
subscriptions, a managed worker, and a middleware hook system. Published on
PyPI as `rele`. Optional Django and Flask integrations.

## Commands

Everything runs through uv (dependencies live in `pyproject.toml`, locked in
`uv.lock` — there are no requirements.txt files):

- `make test` — full suite (pytest via `runtests.py`, needs the django group,
  installed by default with `uv sync`)
- `make lint` — ruff check + ruff format --check
- `make typecheck` — mypy strict over the `rele` package
- `make install` — sync every dependency group and extra

All three of lint/typecheck/test must pass before pushing; CI runs them on
Python 3.10–3.14.

## Hard rules

- **Conventional Commits in PR titles** (`fix:`, `feat:`, `chore:`, ...).
  Squash-merge titles drive release-please version bumps: `fix:` → patch,
  `feat:` → minor, `feat!`/`BREAKING CHANGE` → major.
- **The version lives ONLY in `rele/__init__.py`** (annotated with
  `x-release-please-version`). Never move it to `[project]` in
  pyproject.toml: static versions get recorded in `uv.lock` and every
  release bump would then break `uv sync --locked` in CI.
- **New dotfiles must be un-ignored explicitly**: `.gitignore` has a blanket
  `.*` rule, so `git add` silently skips new dotfiles unless you add a
  `!.yourfile` exception. This has silently broken things before.
- The public API is fully typed and checked with mypy `strict = true`; new
  code must be annotated. `rele/py.typed` ships in the wheel.
- Everything under `rele/` is public API surface — renames and signature
  changes are breaking changes for downstream users.
- CHANGELOG.md is generated by release-please — never edit it by hand.
  CHANGELOG.rst is frozen history.
